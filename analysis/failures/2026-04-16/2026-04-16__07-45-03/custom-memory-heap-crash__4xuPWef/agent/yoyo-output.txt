Both the release and debug builds run successfully with no crashes and no memory leaks. Here's a summary of what was happening and how it was fixed:

## Root Cause Analysis

The modified libstdc++ (in /build/) introduced an MSVC-style facet registration system. In NDEBUG (release) mode, when `__try_use_facet() is called for the first time, it invokes _Facet_Register_impl()` which:
1. Allocates _Fac_node objects via **::operator new**
2. Each node also allocates 256 bytes via `new char[256]`
3. These nodes are stored in a global linked list (_Fac_head) and freed by a static destructor (_Fac_tidy_reg_t) that runs after `main()` returns

The first call to __try_use_facet occurred during `cout << std::fixed << pi_estimate (outputting a double), which was **after g_custom_heap was already active**. This meant the _Fac_node` allocations went into the custom heap arena. When Application::shutdown() destroyed the custom heap (still inside main()), and then the atexit/static destructors ran calling ::operator delete on those nodes — the memory had already been wiped and freed → segfault.

The **debug build** was unaffected because it uses std::malloc/std::free directly (the #ifdef NDEBUG branches).

## Fix (`/app/user.cpp`)

In user_init() (which runs before g_custom_heap is installed), we force all locale facet initialization by:
1. Calling `std::use_facet<> for all common facets with std::locale::classic()`
2. Performing a dummy ostringstream output of a double with std::fixed/std::setprecision

This ensures _Facet_Register_impl() is called — and the _Fac_nodes allocated — while `g_custom_heap` is still null, so they use std::malloc and are safely freed by std::free in the static destructor after main() exits.
