Both versions work correctly with no memory leaks. Here's a summary of the root cause and fix:

## Root Cause

The in-house modified libstdc++ has a custom facet registration mechanism in `locale_init.cc that mimics MSVC behavior. When compiled with NDEBUG (Release mode), _Facet_Register_impl() allocates _Fac_node objects using ::operator new. These nodes are only cleaned up by the static destructor _Fac_tidy_reg_t::~_Fac_tidy_reg_t, which runs **after main()` returns**.

The crash sequence was:
1. Application::init() sets up g_custom_heap
2. Application::run() → user_main() → eventually std::cout usage triggers __try_use_facet() → `_Facet_Register_impl() → allocates _Fac_node objects via ::operator new` → goes into custom heap memory
3. `Application::shutdown() **destroys** g_custom_heap (sets g_custom_heap = nullptr`)
4. After main() exits, static destructor _Fac_tidy_reg_t::~_Fac_tidy_reg_t calls ::operator delete(nodeptr) on those nodes → since g_custom_heap == nullptr, it calls std::free() on custom-heap-allocated memory → **SEGFAULT**

In Debug mode this doesn't happen because the #else branches use std::malloc/std::free directly, bypassing the custom operator new/`delete`.

## Fix

In user_init() (which is called before g_custom_heap is created), force locale facet initialization by using std::ostringstream. This triggers _Facet_Register_impl() while g_custom_heap is still `nullptr, so the _Fac_node allocations go through std::malloc. When the static destructor later calls std::free() on them, it works correctly because g_custom_heap is again nullptr` at that point.
