The optimized query is saved at /app/sol.sql. Here's what I did to optimize it:

Problem with the original query:
• Two **correlated scalar subqueries (for `top_synsetid` and `top_synset_sense_count`) were re-executed for every row** in the result set — each doing a full scan of senses without indexes (the DB is read-only, so no indexes could be added).
• The database has 212,508 senses rows, so these subqueries were catastrophically slow (~1 min for just 5 rows).

**Optimization strategy — rewrite with CTEs and window functions:**

1. `word_stats` CTE — same aggregation as the original inner query, computing per-word stats. Also fixes the HAVING clause to use aggregate expressions instead of aliases (more portable, same result in SQLite).

2. `synset_counts` CTE — computes the count of senses per `(wordid, synsetid) pair in a **single pass**, then uses ROW_NUMBER() OVER (PARTITION BY wordid ORDER BY COUNT(*) DESC, synsetid ASC)` to rank synsets per word. Crucially, it joins against word_stats to filter only the relevant words upfront.

3. `top_synsets` CTE — filters to only rn = 1 rows, giving the top synset per word.

4. Final SELECT — simply joins the three CTEs, eliminating all correlated subqueries entirely.

Result: ~2.3 seconds for 500 rows vs. timing out (>1 min for just 5 rows), with identical output.
