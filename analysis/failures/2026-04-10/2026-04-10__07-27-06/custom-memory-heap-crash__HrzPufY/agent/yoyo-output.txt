## Root Cause Analysis

The custom in-house libstdc++ has a deliberate modification in locale_init.cc that introduces a release-only bug:

**In `_Fac_tidy_reg_t::~_Fac_tidy_reg_t** (runs as an atexit` handler after main() exits):
```cpp
#ifdef NDEBUG
    ::operator delete(nodeptr);  // RELEASE: uses custom operator delete
#else
    std::free(nodeptr);          // DEBUG: uses free directly
#endif
```

**In `_Facet_Register_impl()** (allocates _Fac_node` objects on first locale facet use):
```cpp
#ifdef NDEBUG
    void* mem = ::operator new(sizeof(_Fac_node));  // RELEASE: uses custom operator new
#else
    void* mem = std::malloc(sizeof(_Fac_node));     // DEBUG: uses malloc directly
#endif
```

The crash sequence:
1. `Application::init() sets up g_custom_heap`
2. Later, when a double is printed via std::cout, `_Facet_Register_impl() runs for the first time (it's guarded by _Static_init_complete` so it skips during static init). It allocates 5 _Fac_node objects via ::operator new → which routes to g_custom_heap->allocate() (the arena)
3. `Application::shutdown() destroys the arena and sets g_custom_heap = nullptr`
4. `_Fac_tidy_reg_t::~_Fac_tidy_reg_t() runs as atexit → calls ::operator delete(nodeptr) → g_custom_heap` is null → std::free(arena_ptr) → CRASH

## Fix

Call `std::_Facet_Register_impl() directly from user_init(), which runs **before** g_custom_heap is initialized. This forces the _Fac_node` allocations to happen via std::malloc (since g_custom_heap == nullptr), so `std::free()` during cleanup is valid.

The naive fix of using use_facet<ctype<char>>() doesn't work because `-O2 eliminates the _Facet_Register_impl()` call inside the inlined __try_use_facet<ctype<char>> template — the compiler sees the if constexpr fast-path always taken for ctype<char> and treats the call as dead code.
