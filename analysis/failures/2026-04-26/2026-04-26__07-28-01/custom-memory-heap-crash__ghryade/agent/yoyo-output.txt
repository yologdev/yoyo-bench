Both builds pass cleanly with no memory leaks. Here's a summary of what was found and fixed:

## Root Cause Analysis

The custom in-house libstdc++ at /usr/local/gcc-custom-release/ contains a patched locale_init.cc that adds MSVC-style facet registration. In `RELEASE builds (-DNDEBUG`), the patch:

1. Allocates 5 `_Fac_node objects via ::operator new` and each node allocates a 256-byte buffer via new char[256] — on first call to `_Facet_Register_impl()`.
2. Frees those nodes at program exit via a static destructor (`_Fac_tidy_reg_t::~_Fac_tidy_reg_t()), using ::operator delete`.

The problem: `_Facet_Register_impl() was being called **after** g_custom_heap` was set up (triggered by std::cout << in `Application::init()`). The nodes were therefore allocated **from the custom arena**. When Application::~Application() ran at the end of main(), it destroyed g_custom_heap and freed the arena. Then the static destructor tried to ::operator delete the arena-allocated nodes — but with g_custom_heap == nullptr, that fell through to std::free() on a pointer that was never malloc()'d → segfault.

The DEBUG build didn't crash because it uses `std::malloc/std::free directly instead of ::operator new/::operator delete`.

## The Fix (`/app/user.cpp`)

```cpp
namespace std {
    void _Facet_Register_impl();
}

void user_init() {
    // Force _Facet_Register_impl to run BEFORE g_custom_heap is activated.
    // This ensures the _Fac_node allocations use std::malloc (g_custom_heap==null),
    // so the atexit cleanup via std::free works correctly.
    std::_Facet_Register_impl();
}
```

By calling std::_Facet_Register_impl() in user_init() — which executes before g_custom_heap is set up — the 5 _Fac_node objects and their buffers are allocated via std::malloc. The _Facets_registered flag is then set to true, preventing any re-registration once the custom heap is active. At program exit, `std::free() correctly frees those malloc`-allocated pointers.
