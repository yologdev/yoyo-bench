## Optimization Strategy

Root cause of slowness: The original query had two correlated subqueries in the SELECT clause — one to find top_synsetid and another nested one for `top_synset_sense_count`. These were executed once per qualifying row (thousands of rows), scanning the senses table each time with no indexes.

**Optimizations applied:**

1. Eliminated correlated subqueries — replaced them with CTEs (Common Table Expressions) that compute results in a single pass.

2. Used `ROW_NUMBER()` window function — to efficiently pick the top synset per word (by sense count descending, then synsetid ascending) without re-scanning the table repeatedly.

3. **Structured as a pipeline of CTEs:**
   • base: joins words → senses → synsets once
   • word_stats: aggregates per word with the HAVING filter
   • synset_counts: counts senses per (word, synset) pair, filtered to only qualifying words
   • top_synsets: applies ROW_NUMBER() to rank synsets per word

4. Result: ~82× speedup — from >82 seconds (for just 10 rows) to ~2 seconds for all 500 rows, with identical output.
