## Root Cause & Fix

### The Bug (in the custom libstdc++)

The modified libstdc++ in /build/ contains a fabricated MSVC-style facet registration mechanism in `locale_init.cc. Under NDEBUG (release mode), it allocates _Fac_node tracking objects via ::operator new` / new char[256] when a locale facet is first used, and then frees them via ::operator delete / delete[] inside a static destructor (`_Fac_tidy_reg_t) that runs **after main()` exits**.

The application in main.cpp installs a custom arena allocator (`g_custom_heap) *after* calling user_init() but *before* the first std::cout call. That std::cout triggers __try_use_facet → _Facet_Register_impl(), allocating the _Fac_node`s **from the custom arena**. By the time the static destructor runs post-main, `g_custom_heap is already destroyed and set to null, so operator delete falls through to std::free(arena_ptr)` — a pointer that was never malloc'd → **SIGSEGV**.

In debug mode the library uses `std::malloc/std::free directly (bypassing operator new/delete`) so the same sequence is safe.

### The Fix (`/app/user.cpp`)

Call std::_Facet_Register_impl() **directly* in `user_init()` — which runs before* g_custom_heap is created. At that point, operator new falls through to std::malloc, so the _Fac_nodes are heap-allocated. The _Facets_registered flag is then true, causing all subsequent calls to return immediately without allocating from the custom arena. At program exit, std::free on those malloc'd pointers works correctly.

The direct call (rather than going through use_facet) is essential because `-O2 **inlines** __try_use_facet and **eliminates** the _Facet_Register_impl()` call inside it — a direct external-linkage call cannot be elided by the optimizer.
