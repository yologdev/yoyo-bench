## Root Cause

The custom in-house libstdc++ (in the /build/ patches) introduced an MSVC-style facet registration mechanism that only activates in RELEASE (-DNDEBUG) mode:

1. `_Facet_Register_impl()` — called the first time __try_use_facet is invoked (e.g., when std::cout formats output) — allocates _Fac_node objects and their _Facptr buffers using `::operator new` (in NDEBUG) vs std::malloc (in DEBUG).
2. These nodes are stored in a global linked list (`_Fac_head) and freed by a static destructor (_Fac_tidy_guard`) that runs after `main()` returns.
3. main.cpp overrides `operator new/delete to use a custom heap (g_custom_heap`) when it's active.

The crash sequence in RELEASE:
• `app.init() calls user_init() (custom heap not yet active), then **creates** g_custom_heap`.
• app.run() uses std::cout → triggers __try_use_facet → _Facet_Register_impl() → allocates nodes with ::operator new → goes to the custom heap.
• app.shutdown() destroys g_custom_heap (sets it to nullptr).
• After main() returns, _Fac_tidy_guard::~_Fac_tidy_guard() calls `::operator delete(nodeptr) → g_custom_heap is nullptr → calls std::free()` on a pointer that was **never from malloc → segfault**.

## The Fix (`user.cpp`)

Call std::cout.imbue(std::locale::classic()) inside user_init(), which is called before g_custom_heap is created. This forces __try_use_facet → _Facet_Register_impl() to fire while g_custom_heap is still nullptr, so all facet nodes are allocated via the system malloc/free. By the time `_Fac_tidy_guard cleans them up after main() exits, g_custom_heap is again nullptr, and std::free()` correctly frees the malloc-allocated nodes without crashing.
