## Analysis of the Problem

The original query was extremely slow due to correlated subqueries in the SELECT clause:
1. top_synsetid — a subquery that runs for every row in the outer result
2. top_synset_sense_count — a subquery containing another nested subquery (triple nesting!) that also runs per row

These correlated subqueries caused repeated full scans of the senses table for each of the 500+ result rows.

## Optimization Strategy

I rewrote the query using CTEs and window functions to eliminate all correlated subqueries:

1. `base` CTE — joins words, senses, and `synsets` once
2. `agg` CTE — computes all aggregate stats (total_synsets, total_senses, distinct_domains, distinct_posids) with the HAVING filter, replacing the inline subquery
3. `synset_counts` CTE — precomputes per-word per-synset sense counts in a single pass, filtered to only qualifying words
4. `top_synset` CTE — uses ROW_NUMBER() OVER (PARTITION BY wordid ORDER BY cnt DESC, synsetid ASC) to find the top synset per word in one pass — replacing the two deeply-nested correlated subqueries

Result: Query runs in ~4 seconds instead of timing out (>2 minutes), producing the same 500 rows with identical output.
