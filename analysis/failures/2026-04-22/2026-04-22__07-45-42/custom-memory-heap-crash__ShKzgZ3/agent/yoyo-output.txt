────────────────────────────────────────

## Root Cause

The custom in-house libstdc++ (release variant, compiled with -DNDEBUG) contains a modified `locale_init.cc` that mimics MSVC's facet-registration pattern. The key code path:

1. **_Facet_Register_impl()** — called the first time any locale-dependent operation (e.g. number formatting via `__try_use_facet) runs. In NDEBUG mode it allocates _Fac_node objects and their inner _Facptr buffers using **::operator new** / **new char[256]`**.

2. `_Fac_tidy_reg_t::~_Fac_tidy_reg_t()` — a static destructor that fires after `main()` exits and frees those nodes via `::operator delete`.

The bug: main.cpp's Application::init() activates g_custom_heap before the first `std::cout << call. So when std::cout << "=== Monte Carlo..."` triggers _Facet_Register_impl(), the facet nodes land in the custom arena (not the regular heap). Later, Application::shutdown() destroys `g_custom_heap and sets it to nullptr *before* main()` returns. When the static destructor then calls ::operator delete(nodeptr), `g_custom_heap is already nullptr, so it falls back to std::free()` — which crashes on arena memory.

────────────────────────────────────────

## Fix (`user.cpp`)

```cpp
void user_init() {
    // Trigger locale/facet one-time registration HERE, before main.cpp
    // activates g_custom_heap, so the _Fac_node allocations go through
    // plain std::malloc instead of the custom arena.
    std::ostringstream oss;
    oss << 0;
    (void)oss.str();
}
```

By performing a locale-dependent numeric formatting operation in user_init() (which runs before g_custom_heap is set up), the _Facets_registered flag is set to true and the nodes are allocated with plain `malloc. When the static destructor later calls ::operator delete → std::free`, it works correctly, and the crash is gone.
