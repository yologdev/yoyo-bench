## Regex Structure

```
^
(?=[^\n]*  ← lookahead: line contains a valid IPv4
  (?<![a-zA-Z0-9])  ← IPv4 not preceded by alphanumeric
  (?:25[0-5]|2[0-4][0-9]|1[0-9]{2}|[1-9][0-9]|[0-9])  ← first octet (no leading zeros)
  (?:\.(?:25[0-5]|2[0-4][0-9]|1[0-9]{2}|[1-9][0-9]|[0-9])){3}  ← 3 more octets
  (?![a-zA-Z0-9])  ← IPv4 not followed by alphanumeric
)
[^\n]*  ← consume chars in line up to the date
(?<![a-zA-Z0-9])  ← date not preceded by alphanumeric
(  ← CAPTURE the date
  \d{4}-(?:
    (?:0[13578]|1[02])-(?:0[1-9]|[12][0-9]|3[01])  ← 31-day months
    | (?:0[469]|11)-(?:0[1-9]|[12][0-9]|30)         ← 30-day months
    | 02-(?:0[1-9]|1[0-9]|2[0-9])                   ← Feb (01–29)
  )
)
(?![a-zA-Z0-9])  ← date not followed by alphanumeric
(?![^\n]*  ← negative lookahead: no more valid dates after this one in the line
  (?<![a-zA-Z0-9])\d{4}-...(same date pattern)...(?![a-zA-Z0-9])
)
```

Key design decisions:
• [^\n]* (not .*) ensures all matching stays within a single line
• The greedy [^\n]* before the captured date naturally advances to the last possible date (combined with the negative lookahead that rejects matches when more dates follow)
• IPv4 validation excludes leading zeros and values >255 per octet
• Date validation handles month-specific day limits (29 for Feb, 30 for Apr/Jun/Sep/Nov, 31 for Jan/Mar/May/Jul/Aug/Oct/Dec)
