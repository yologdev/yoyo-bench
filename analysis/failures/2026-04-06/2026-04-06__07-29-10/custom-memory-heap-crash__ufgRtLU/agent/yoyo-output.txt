Both debug and release builds work correctly with no memory leaks. Here's a summary of what was found and fixed:

## Root Cause Analysis

The crash in RELEASE mode was caused by a complex chain of events:

1. Custom heap manager in main.cpp replaces operator new/delete globally. When g_custom_heap is active, allocations go into a 10MB bump-allocator; when it's null, they use `malloc/free`.

2. In-house libstdc++ has a modified locale_init.cc that (in #ifdef NDEBUG / release mode) registers locale facets via ::operator new and cleans them up via ::operator delete in a static destructor (_Fac_tidy_reg_t).

3. The trigger: _Facet_Register_impl() (which allocates the _Fac_nodes) is guarded by _Static_init_complete — only running after static initialization finishes. The first call that passes this guard happens during Application::run() when printing a `double value (cout << pi_estimate), **at which point g_custom_heap` is already active**. So the nodes are allocated from the custom heap.

4. **The crash**: Application::~Application() destroys and frees the custom heap (poisoning its memory with 0xEF). Then the static _Fac_tidy_reg_t destructor runs, calls ::operator delete(nodeptr) where nodeptr is inside the already-freed 10MB block → segfault.

5. **Why use_facet in user_init() didn't help**: At -O2, the compiler inlines `__try_use_facet<ctype<char>>. Since ctype<char>` is a standard facet resolved via a compile-time if constexpr, the compiler determines the `_Facet_Register_impl()` call has no observable effect on the result and **eliminates it entirely**.

## The Fix

Call std::_Facet_Register_impl() directly in `user_init(), which runs before g_custom_heap` is set up. This forces allocation of the _Fac_nodes via malloc, so when the static destructor later frees them (with g_custom_heap == nullptr), it safely calls std::free() on properly malloc-allocated memory.
