## Optimizations Applied

**Problem with the original query: It used two deeply nested correlated subqueries** in the SELECT clause — one to find top_synsetid and one to count top_synset_sense_count. Each row in the result required executing these subqueries (which themselves contained a nested correlated subquery each), causing the original to be far too slow to finish.

**Solution — using CTEs and a window function:**

1. `base` CTE: Does the three-table join once, materializing the data needed for all aggregations.

2. `agg` CTE: Computes the per-word aggregate stats (total_synsets, total_senses, `distinct_domains, distinct_posids) with HAVING` filters — replacing the inner subquery.

3. **synset_counts CTE: Pre-computes sense counts per `(wordid, synsetid)` pair only for words that passed the filter**, eliminating the need to re-scan the senses table repeatedly.

4. `top_synsets` CTE: Uses ROW_NUMBER() window function to pick the top synset per word in a single pass, completely replacing the two correlated subqueries (which each ran O(N) subqueries).

5. Final join: Simple JOIN between agg and `top_synsets, with the correct ORDER BY and LIMIT 500`.

Result: Drops runtime from >2 minutes (timeout) down to ~3.4 seconds, producing identical output.
