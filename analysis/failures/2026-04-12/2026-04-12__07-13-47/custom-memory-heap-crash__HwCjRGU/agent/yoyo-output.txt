All checks pass. Here's a summary of what was happening and how it was fixed:

## Root Cause

The in-house libstdc++ release build has a modified locale_init.cc with `#ifdef NDEBUG` conditional behavior:

• In RELEASE (`-DNDEBUG): _Fac_node` objects and their data are allocated with ::operator new and freed with ::operator delete / `delete[]`
• In DEBUG: they use std::malloc / std::free directly

The crash sequence:
1. Application::init() calls user_init() first, **then sets up `g_custom_heap`, then** calls cout <<
2. That first cout << operation triggers locale/facet initialization (_Facet_Register_impl()), which allocates `_Fac_node objects using ::operator new — **but g_custom_heap` is now active**, so they land in the custom 10 MB heap buffer
3. When main() returns, Application::~Application() destroys g_custom_heap and sets it to nullptr
4. Static destructors then run: `_Fac_tidy_reg_t::~_Fac_tidy_reg_t() calls ::operator delete(nodeptr)` — with g_custom_heap == nullptr, this falls through to std::free(nodeptr), but `nodeptr` points into the middle of the already-freed 10 MB buffer → segfault

## Fix (`/app/user.cpp`)

In user_init() (which is called before g_custom_heap is set up), force locale initialization to happen while g_custom_heap is still nullptr:

```cpp
void user_init() {
    // Force locale/facet initialization BEFORE the custom heap manager is set up.
    std::cout.imbue(std::locale::classic());
}
```

This causes _Facet_Register_impl() to run while g_custom_heap == nullptr, so the _Fac_node allocations use plain std::malloc. When static destructors later call std::free() on them, it works correctly — no crash, no leaks.
