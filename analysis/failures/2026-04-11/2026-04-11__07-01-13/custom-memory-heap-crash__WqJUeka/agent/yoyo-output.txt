────────────────────────────────────────

## Root Cause

The custom `libstdc++ at /build/` contains a modified locale_init.cc that mimics an MSVC-style "facet registration" system. When __try_use_facet() is first called (which happens on any formatted I/O operation like `std::cout << ...), it calls _Facet_Register_impl()`, which allocates a linked list of _Fac_node objects using `::operator new` (in `NDEBUG`/release mode).

The crash sequence in RELEASE mode was:

1. Application::init() calls user_init() *(nothing happens)*, then **activates `g_custom_heap`**.
2. Application::run() uses std::cout → triggers __try_use_facet() → _Facet_Register_impl() runs → _Fac_node objects are allocated via ::operator new → **routed to the custom arena heap** (because g_custom_heap != nullptr).
3. `Application::shutdown()` destroys the arena: wipes the 10 MB block with 0xEF, then calls std::free on it. g_custom_heap is set to nullptr.
4. Static destructors run → `_Fac_tidy_reg_t::~_Fac_tidy_reg_t() calls ::operator delete(nodeptr) on each _Fac_node. Now g_custom_heap is null`, so it falls through to std::free() — but that memory belongs to the already-freed arena → SIGSEGV.

In DEBUG mode it used std::malloc/std::free directly (bypassing the overloaded `operator new/delete`), so no crash.

## Fix (`/app/user.cpp`)

Force `_Facet_Register_impl() to run **inside user_init()`**, which is called before g_custom_heap is set up in Application::init():

```cpp
void user_init() {
    std::ostringstream oss;
    oss << 0;   // triggers __try_use_facet → _Facet_Register_impl()
    (void)oss.str();
}
```

This ensures the _Fac_node objects are allocated when g_custom_heap is still nullptr, so plain std::malloc is used. The subsequent std::free calls in the static destructor succeed correctly, and the `_Facets_registered` flag prevents any re-allocation once the custom heap becomes active.
