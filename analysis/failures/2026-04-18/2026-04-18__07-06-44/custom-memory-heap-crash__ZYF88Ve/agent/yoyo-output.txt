Both release and debug builds pass with zero memory leaks. Here's a summary of what was wrong and how it was fixed:

## Root Cause

The in-house modified libstdc++ in `/build/gcc-13.2.0/libstdc++-v3/src/c++98/locale_init.cc` contains a custom facet-tracking mechanism that — under `NDEBUG` (release) only — allocates `_Fac_node objects using ::operator new when __try_use_facet()` is first called (i.e., on the first locale-aware I/O operation). A static destructor (_Fac_tidy_reg_t) then cleans up these nodes after main() exits.

The crash sequence was:

1. `app.init() sets up g_custom_heap, then calls std::cout << "=== Monte Carlo..."` — this is the **first** formatted I/O, which triggers _Facet_Register_impl() → ::operator new → custom heap allocates _Fac_node objects inside the custom heap's 10 MB buffer.
2. app.~Application() → shutdown() destroys the custom heap: `memset(0xEF, ...) + std::free(). The _Fac_node` list is now dangling pointers into freed memory.
3. After main() returns, the static _Fac_tidy_reg_t destructor runs and tries to walk _Fac_head — crash accessing already-freed (and poisoned) memory.

In debug mode the same code uses std::malloc for the allocations (not ::operator new), so they live outside the custom heap and survive it.

## Fix (`/app/user.cpp`)

```cpp
void user_init() {
    // Trigger locale facet initialization (via std::locale::classic()) BEFORE
    // the custom heap is activated. This causes _Facet_Register_impl() to run
    // while g_custom_heap == nullptr, so _Fac_nodes are allocated with plain
    // std::malloc. The _Fac_tidy_reg_t static destructor can then safely
    // std::free() them after main() exits.
    std::cout.imbue(std::locale::classic());
}
```

user_init() is called before g_custom_heap is set, so calling std::cout.imbue(std::locale::classic()) forces locale facet registration at a point where g_custom_heap == nullptr. The _Fac_node allocations therefore use std::malloc, and the static destructor's std::free() calls succeed cleanly.
