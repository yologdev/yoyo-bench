## Root Cause

The modified libstdc++ at /build/ has an intentional RELEASE-mode bug in its locale implementation (`locale_init.cc). The patch introduces a linked list of _Fac_node objects (mimicking MSVC's _Fac_tidy_reg_t` pattern) that are:

• **Allocated** via ::operator new (RELEASE only) inside _Facet_Register_impl(), which is called on **first use** of any locale facet (e.g., the first std::cout << call).
• Freed via ::operator delete inside a **static destructor** _Fac_tidy_reg_t::~_Fac_tidy_reg_t() — which runs after `main()` returns.

The crash chain in the original code:

1. app.init() installs g_custom_heap (a bump allocator), then calls `std::cout <<` for the first time.
2. std::cout << triggers `__try_use_facet → _Facet_Register_impl() → allocates _Fac_node objects via ::operator new` → they land in the bump allocator.
3. `app.run() finishes; ~Application() calls shutdown()`, which destroys g_custom_heap and sets it to nullptr.
4. After `main()` returns, the static destructor _Fac_tidy_reg_t::~_Fac_tidy_reg_t() calls `::operator delete() on those nodes. Since g_custom_heap == nullptr` now, it calls std::free() on pointers that came from the bump allocator → SIGSEGV.

## The Fix

In `user_init() — which is called **before** g_custom_heap` is installed — we force locale initialization:

```cpp
void user_init() {
    std::locale loc;      // initialises the classic locale
    std::cout.imbue(loc); // triggers __try_use_facet → _Facet_Register_impl
}
```

This causes _Facet_Register_impl() to run while g_custom_heap is still nullptr, so all _Fac_node allocations go through plain std::malloc. When the static destructor later calls ::operator delete() on them (after main(), with g_custom_heap == nullptr), it correctly calls std::free() — no crash, no leak.
