## The Bug

The issue was in the pool_sweep function in `/app/ocaml/runtime/shared_heap.c`.

The run-length compression scheme encodes free blocks so that `POOL_FREE_HEADER(n) marks a free slot whose wosize` field stores n — meaning "there are n more contiguous free slots immediately following me." Each slot in a pool of sizeclass sz is exactly `wh = wsize_sizeclass[sz]` words wide (including the header word).

### The broken code (line 650):
```c
p += wh * Wosize_hd(hd);   // skip the n following free slots
// ...
p += Whsize_hd(hd);         // ← BUG: Whsize_hd = Wosize_hd + 1 = n + 1, not wh
```

For a free block, Wosize_hd(hd) = n (run-length count), so Whsize_hd(hd) = n + 1. But the current slot always occupies exactly wh words. So instead of advancing `wh words past the current slot, it advanced only n + 1 words — completely wrong for any wh > 1 or n > 0. This caused p` to point at garbage mid-slot data, corrupting the sweep and causing the compiler to crash during bootstrapping.

The same p += wh pattern is used correctly in every other pool-iteration loop in the file (calc_pool_stats, pool_finalise, `caml_redarken_pool`, etc.).

### The fix:
```c
p += wh * Wosize_hd(hd);   // skip the n following free slots
// ...
p += wh;                    // ← FIXED: advance exactly one slot width
```
