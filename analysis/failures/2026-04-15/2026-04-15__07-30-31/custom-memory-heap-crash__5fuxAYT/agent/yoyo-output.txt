All checks pass. Here's a summary of what was happening and how it was fixed:

## Root Cause

The in-house libstdc++ in /build/ was patched with a "MSVC-style facet registration" mechanism that allocates `_Fac_node objects and stores them in a linked list (_Fac_head`), cleaned up after main() exits via a static destructor (_Fac_tidy_reg_t).

The critical difference between DEBUG and RELEASE builds (controlled by `#ifdef NDEBUG`):
• RELEASE: Uses ::operator new / ::operator delete for _Fac_node allocations
• DEBUG: Uses std::malloc / std::free directly

The crash sequence in RELEASE:
1. `app.init() sets up g_custom_heap` (a 10MB arena allocator), then uses std::cout 
2. std::cout triggers __try_use_facet → _Facet_Register_impl() → allocates 5 _Fac_node objects via ::operator new → goes into the custom arena
3. `~Application() destroys g_custom_heap`, sets it to nullptr, then frees the 10MB arena block
4. After main() returns, _Fac_tidy_reg_t::~_Fac_tidy_reg_t() runs (static destructor) → calls `::operator delete(nodeptr)` on those nodes
5. g_custom_heap is nullptr → falls through to `std::free(arena_pointer)` → SIGSEGV

## Fix (`/app/user.cpp`)

In user_init() — which is called before g_custom_heap is installed — force locale facet initialization by performing a stream operation (e.g., oss << 0.0). This triggers __try_use_facet → `_Facet_Register_impl() while g_custom_heap is still nullptr, so the 5 _Fac_node objects are allocated via std::malloc`. Later, when static destructors call std::free() on those nodes, it's completely safe.
