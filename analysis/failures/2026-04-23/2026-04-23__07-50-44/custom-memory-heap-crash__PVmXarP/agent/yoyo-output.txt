All checks pass. Here's a summary of what was found and fixed:

## Root Cause

The custom in-house libstdc++ has an intentional/buggy difference between DEBUG and RELEASE builds in locale_init.cc. The `_Facet_Register_impl() function allocates _Fac_node` objects to track locale facets for cleanup at program exit:

• DEBUG: allocates with std::malloc / frees with std::free  
• RELEASE (-DNDEBUG): allocates with ::operator new / frees with `::operator delete`

The crash occurs because:
1. main() activates the `CustomHeapManager (a bump allocator) which intercepts ::operator new`
2. When `cout prints the double value (π estimate), num_put<char>::_M_insert_float calls __try_use_facet → _Facet_Register_impl() — allocating 5 _Fac_node` objects **from the custom heap buffer**
3. Application::shutdown() destroys the CustomHeapManager, poisons its memory with 0xEF, and frees the backing buffer
4. At program exit, the static _Fac_tidy_guard destructor tries to free those nodes — but g_custom_heap is now nullptr, so it falls through to std::free() on pointers into the already-freed/poisoned buffer → SIGSEGV

## Fix

In user_init() (which runs before the custom heap is activated), call std::_Facet_Register_impl() directly. This pre-registers all locale facet nodes using the standard malloc allocator while g_custom_heap is nullptr, so the static destructor can safely free them with std::free at exit.

A simple `use_facet<ctype<char>>() call wasn't sufficient because the optimizer inlines __try_use_facet<ctype<char>>` and eliminates the _Facet_Register_impl() call entirely (since `ctype<char> is always present in std::locale). Calling _Facet_Register_impl()` directly bypasses this optimization.
