Both builds run cleanly with zero errors and zero memory leaks. Here's what the root cause was and how the fix works:

## Root Cause Analysis

The custom in-house libstdc++ has an intentional (buggy) modification in locale_init.cc that mimics MSVC-style facet registration. When compiled with -DNDEBUG (RELEASE mode), it:

1. Allocates _Fac_node objects (and their 256-byte _Facptr buffers) using ::operator new when locale facets are **first used** at runtime (_Facet_Register_impl).
2. Frees them using ::operator delete in the destructor of a static _Fac_tidy_guard object, which runs after `main()` exits.

The crash sequence was:
1. init() creates g_custom_heap (the arena allocator)
2. `std::cout << "=== Monte Carlo..."` — first use of cout triggers locale facet lookup → _Facet_Register_impl() runs → `_Fac_node objects allocated via ::operator new` → **routed into the arena heap**
3. ~Application() → shutdown() → destroys g_custom_heap, sets it to `nullptr`
4. After main() returns, _Fac_tidy_guard::~_Fac_tidy_guard() calls ::operator delete on the `_Fac_node pointers → g_custom_heap is nullptr → falls through to std::free()` on **arena memory → SEGFAULT**

## The Fix (`user.cpp`)

user_init() is called before g_custom_heap is set up. By forcing locale facet initialization there (via std::use_facet<std::ctype<char>> and std::ostringstream), _Facet_Register_impl() runs while g_custom_heap == nullptr. This means the _Fac_node allocations go through std::malloc, and the corresponding std::free calls at cleanup are perfectly safe.
