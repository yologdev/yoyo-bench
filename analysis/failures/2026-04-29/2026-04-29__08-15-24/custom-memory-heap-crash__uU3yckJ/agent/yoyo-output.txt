Both builds work correctly and Valgrind reports zero memory leaks. Here's what I found and fixed:

## Root Cause

The patched libstdc++ (found in `/build/patches/`) contains a deliberate modification that behaves differently based on NDEBUG:

• DEBUG mode: Locale facet registrations use std::malloc/std::free directly
• RELEASE mode (-DNDEBUG): Locale facet registrations use ::operator new/::operator delete

The main.cpp installs a **custom heap manager** (g_custom_heap) that intercepts operator new/delete. The crash sequence was:

1. Application::init() activates g_custom_heap
2. The first std::cout << triggers locale facet initialization → those facets are allocated via ::operator new → routed into the custom heap arena
3. Application::shutdown() destroys g_custom_heap and sets it to nullptr
4. After `main()` returns, static destructors clean up locale facets via ::operator delete → with g_custom_heap == nullptr, this calls std::free() on a pointer that was never malloc'd (it was from the arena) → SEGFAULT

## Fix (in `user.cpp`)

Force locale and facet initialization before the custom heap becomes active (i.e., inside user_init(), which is called first in `Application::init()`):

```cpp
void user_init() {
    // Force locale initialization before the custom heap manager is activated.
    // In RELEASE mode, the patched libstdc++ allocates locale facets via
    // ::operator new (routed through custom heap when active). Those facets
    // are freed during static destruction after main() returns — by which
    // point the custom heap is already destroyed, causing a crash.
    // Triggering facet initialization here ensures they use the real malloc.
    std::locale loc;
    (void)std::use_facet<std::ctype<char>>(loc);
    std::cout.sync_with_stdio(false);
}
```
