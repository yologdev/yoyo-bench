Both debug and release now work perfectly with zero memory leaks. Here's what I found and fixed:

## Root Cause

The crash was caused by a subtle interaction between the custom heap manager in `main.cpp` and lazy locale initialization in the C++ standard library.

Here's the sequence that caused the crash:

1. app.init() calls user_init() first, **then** sets up g_custom_heap
2. The first `std::cout with floating-point output (inside Application::run()`) triggers lazy initialization of locale facets (e.g., num_put, num_get) — this happens *after* g_custom_heap is active
3. These locale facet objects are allocated via our overridden operator new, which routes them into the custom pool (not real heap memory)
4. The locale facets are registered with atexit() for cleanup at program exit
5. At program exit, `~_Fac_tidy_reg_t tries to delete these objects — but by now g_custom_heap is nullptr, so operator delete falls back to std::free() on **pool addresses that were never malloc`'d → SEGFAULT**

In debug mode, the debug libstdc++ apparently initializes locale facets earlier (before g_custom_heap is set) or handles deallocation differently, masking the bug.

## Fix (`/app/user.cpp`)

In user_init() (called before g_custom_heap is set up), I force the locale facets to initialize by formatting a floating-point number through an ostringstream. This causes all locale allocations to happen via regular malloc/free, so the atexit cleanup works correctly:

```cpp
void user_init() {
    // Force locale facet initialization before custom heap is active
    std::ostringstream oss;
    oss << std::fixed << std::setprecision(2) << 3.14;
    (void)oss.str();
}
```
